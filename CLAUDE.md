# pay_limit_management — указания для реализации

Прочти сначала `../CLAUDE.md` (общие правила, доступ к wiki). Спецификация сервиса — wiki
`1f9a4b6c-9ff2-4a09-b510-061838e70d7d`, план реализации — `11d1f8de-9914-4339-a8c5-6a75df299052`.
Доменная модель (типы контролей, модель правила, инвариант) — техспека §2, здесь не дублируется.

## Что делает сервис

Администрирование лимитов: справочники, правила, группы/членства, назначения трёх уровней
(GLOBAL/GROUP/MERCHANT); компиляция иммутабельных версионированных runtime-манифестов для engine.
PostgreSQL `limit_management`, REST `/internal/v1/limit-management/**` (только через payadmin-bff).

## Порядок этапов (детали и DoD — в плане реализации)

1. Справочники и модель правила (меры PER_OPERATION/INTERVAL, aggregationScope,
   errorMessageTemplate, operation_type + direction + counterparty_type, валидации 1–4)
2. Уровень GLOBAL в назначениях
3. Инвариант непересечения видов лимитов (три точки проверки + advisory lock + 409 conflicts)
4. Манифест v2 (schemaVersion, канонизация, checksum, 422, If-None-Match) → **веха M1**
5. Аудит (X-Operator-Id, audit_event в транзакции)
6. effective-limits + метрики → **веха M2**
7. Сид первой группы (DRAFT, активация вручную)

## Ловушки и жёсткие требования

- **Инвариант непересечения** НЕ выражается констрейнтом БД (три таблицы) — только
  application-проверка под advisory lock (по merchant_id для членств, по rule_id для
  назначений/активации) + повторная проверка при компиляции. Не пытайся сделать триггером.
- **Канонизация манифеста**: детерминированная сортировка (rules по code+version, остальное
  по UUID), отсортированные ключи JSON, без пробелов, суммы строками. Тест на стабильность
  checksum при переупорядоченном входе обязателен (MGT-U-06/07). Любое изменение алгоритма =
  инкремент schemaVersion.
- **Публикация манифеста** — LOCK TABLE ... IN EXCLUSIVE MODE, версия = max+1, заголовок +
  тело + построчные срезы в одной транзакции. effectiveFrom в прошлом → 400.
- **Rollback манифеста** = новая версия с содержимым старой (история линейна), не удаление.
- **Аудит** пишется в той же транзакции, что и изменение; мутация без X-Operator-Id → 400.
- DTO `conflicts` (409/422) — один и тот же формат для всех трёх точек инварианта (пример
  в спеке §3.4).
- Существующие эндпоинты и поведение (CRUD, lifecycle, GiST-инварианты периодов) не ломать —
  они используются как есть.
